---
name: commit-message
description: Generate a commit message based on the code changes.
---

You are responsible for generating clear, concise, and meaningful commit messages based on code changes. When generating a commit message:

1. Analyze the code changes by reviewing the git diff, modified files, implementation details, and intent of the changes.

2. Follow the Conventional Commits specification for structuring the commit message.

3. Use the appropriate commit type based on the changes. Examples include:
   - `feat`: A new feature
   - `fix`: A bug fix
   - `docs`: Documentation-only changes
   - `style`: Formatting, whitespace, or non-functional code style updates
   - `refactor`: Code restructuring without behavior changes
   - `perf`: Performance improvements
   - `test`: Adding or updating tests
   - `build`: Build system or dependency changes
   - `ci`: CI/CD configuration changes
   - `chore`: Maintenance tasks
   - `revert`: Reverting a previous commit

4. Use the proper Conventional Commit format:

```
<type>(optional-scope): short summary

Optional body:
- Explain what changed
- Explain why the change was necessary
- Include relevant implementation details when helpful

Optional footer:
- Reference breaking changes using: BREAKING CHANGE: description
- Include issue references when applicable
```

5. Keep commit messages:
   - Clear and concise
   - Specific to the actual code changes
   - Written in imperative mood (e.g., "add authentication" instead of "added authentication")
   - Lowercase where appropriate for consistency

6. Infer the most appropriate commit type and optional scope from the changed files and implementation details.

7. Do not invent changes, motivations, or context that are not evident from the code changes.

8. Commit granularity — do not create a single umbrella commit covering all changes. Instead:
   - Create one commit per file change, with a message specific to that file's change.
   - Group multiple files into a single commit ONLY when they form one logical, inseparable change (e.g., a generated/output folder such as an Allure report folder containing thousands of files that change together, or auto-generated artifacts that must move as a unit).
   - When grouping is unavoidable, use a scope that reflects the folder or artifact (e.g., `chore(allure-report): regenerate report output`) rather than listing individual files.
   - Prefer the smallest meaningful unit of change so history stays granular and reviewable.

9. Never add Claude, Anthropic, ChatGPT, Copilot, AI assistants, or any automated system as:
   - Contributor
   - Co-author
   - Commit author
   - Reviewer
   - Attribution footer

10. Do not include phrases like:
    - "Generated by Claude"
    - "Co-authored-by: Claude"
    - "AI-assisted"
    - "Generated with ChatGPT"
    - Any attribution to AI systems

11. Output only the final commit message in valid Conventional Commit format. Examples:

```
feat(auth): add JWT refresh token support
fix(invoice): prevent duplicate invoice creation
refactor(api): simplify booking validation logic
docs(readme): update installation instructions
```
